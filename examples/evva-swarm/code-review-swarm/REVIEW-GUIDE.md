# 審查面向定義(公開知識,所有成員可查閱)

三位審查者各自負責一個面向,**只報自己面向的發現**;面向間若有重疊
(例如一個邏輯錯誤同時是安全漏洞),由先發現者報,lead 彙整時去重。

## 正確性(reviewer-correctness)

- 邏輯錯誤:條件寫反、邊界(off-by-one、空集合、nil/None)、提前 return 漏路徑。
- 並發問題:race、未保護的共享狀態、忘記解鎖/關閉。
- 錯誤處理:吞掉的 error、錯誤路徑下的資源洩漏、不一致的錯誤語意。
- 與既有行為的矛盾:改了 A 卻沒改依賴 A 的 B、破壞既有測試所保護的契約。

## 安全(reviewer-security)

- 注入:SQL / 命令 / 路徑拼接、未轉義的輸出。
- 機密:硬編碼的金鑰、token、密碼;機密寫進 log。
- 權限與驗證:缺少的存取檢查、信任了不可信輸入(反序列化、檔案上傳)。
- 依賴與設定:已知有漏洞的依賴版本、過寬的 CORS / 權限設定。

## 品質(reviewer-quality)

- 重複:已有現成函式/工具卻重寫一份。
- 過度複雜:可以三行寫完的十五行、不必要的抽象層。
- 效能:迴圈裡的重複查詢/IO、明顯的 O(n²) 可避免處。
- 可讀性:誤導性命名、與周圍程式碼風格不一致、註解與程式碼不符。

## 發現的格式(三位審查者共用)

每條發現寫成:

```markdown
## [面向-編號] 一句話標題
- 位置: 檔案路徑:行號
- 嚴重度: critical / major / minor
- 說明: 問題是什麼、為什麼是問題(引用相關程式碼)
- 建議: 怎麼改(一兩句即可,不用寫完整 patch)
```

## 嚴重度定義

- **critical**:會造成資料損壞、安全漏洞、或正常使用路徑必然出錯。
- **major**:特定條件下出錯、或顯著的品質/效能問題。
- **minor**:不影響正確性的改善建議。

## 對抗驗證(verifier)

每條發現都會交給 verifier 反駁:verifier 親自重讀相關程式碼,**以推翻
發現為目標** — 找出發現者漏看的防護、誤讀的控制流、或實際上不可達的
路徑。推翻不了的發現才算成立。判定只有三種:

- **confirmed**:重讀後問題確實存在,附上自己的證據。
- **refuted**:發現不成立,附上推翻的理由(漏看了哪行、哪個檢查)。
- **uncertain**:無法確定(例如需要執行期資訊),附上說明 — 報告中如實標註。
